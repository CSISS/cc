1. Brief introduction

CyberConnector, as an EarthCube building block, is developed for facilitating the automatic preparation and feeding of both historic and near-real time Earth Observation (EO) customized data and on-demand derived products into Earth science models (ESM).


It will automatically process the EO data into the right products in the right form needed for ESM initialization, validation, and inter-comparison. It can support many different ESMs through its standard interfaces under a unified framework.


The goals of CyberConnector are:

  • Efficiently and robustly preparing and feeding EO data to ESMs
  • Efficiency, scalability, and generality
  • Enabling the knowledge accumulation and sharing
  • Automatically presenting ESM outputs/results through the Web
  • Facilitating Business Event Processing (BEP)
  • Engaging in broad community participations in developing and using CyberConnector and sustaining its operation.

2. Framework

The framework of CyberConnector is designed by following the spirits and conceptual rules of model web. It contains several modules most of which are distributable and selfrunnable. The characteristics make CyberConnector into a very loosely coupled and scalable framework. Simply speaking, CyberConnector is composed of five major components: model designer, model instantiation, model runner, model monitor and order manager.

  • Model designer helps experts to design abstract models based on their specific domain knowledge.
  • Model instantiation is responsible to instantiate the designed models into runnable workflow instances.
  • Model runner is used to deploy and execute the runnable workflow by user-specified inputs.
  • During the execution, model monitor will dynamically capture the status of each model run.
  • Order manager is responsible to interact with end-point users to notify them what status their orders are in.

3. Preprocess models

CyberConnector provides and regularly maintains the preprocess models of three science models for demonstrating the feasibility of the CyberConnector idea. The three default supported models are:

  • Finite Volume Community Ocean Model (FVCOM)
    • model introduction: FVCOM is a prognostic, unstructured-grid, finite-volume, free-surface, three-dimensional primitive equation coastal ocean circulation model. FVCOM solves controlling partial differential equations using the finite-volume method. FVCOM is solved numerically by flux calculation using the integral form of the governing equations over a horizontal unstructured triangular grid and a vertical terrain-following coordinate. It provides perfect local conservation of mass, momentum, salt, heat, and tracer. These features make FVCOM ideally suited for environmental prediction and interdisciplinary applications in coastal oceans and estuaries.
    • The preprocess models for FVCOM include:
      • Temperature and salinity condition
        • explanation: Initial fields of water temperature and salinity at individual triangular nodes. This file is extracted and interpolated from HYCOM output (ftp://ftp.hycom.org/datasets/ GOMl0.04 /expt_31.0/data/) which is in NETCDF format (Filename example: archv.2010_001_00_3z.nc). Steps to process this data:
          • Read NETCDF files and interpolate T and S at individual nodes;
          • Rewrite the output to FVCOM input format.
        • model graph: View
        • coverage: every hour between 2009.091 and 2014.365 (julian day format)
        • average duration time of model run: 12 hours
      • River discharge
        • explanation: River discharge data including number of rivers, discharge volume, discharge temperature, and discharge salinity. River discharge data are extracted from USGS website. Details are introduced in the fvcom page.
        • model graph: View
        • coverage: yearly product, from 2010 to 2014
        • average duration time of model run: 30 minutes
      • Wind forcing
        • explanation: The real-time fields of wind velocity or wind stress used for FVCOM. Wind data comes from North American Regional Reanalysis (NARR, ftp://nomads.ncdc.noaa.gov/NARR/), which is 3-hourly and in the GRB format. Steps to convert NARR to FVCOM input file: ï‚·
          • Download data from NARR with python script and convert GRB format to txt;
          • Pick up data within a small area which you are interested in;
          • Interpolate wind velocity components u/v at each FVCOM triangular centroids;
          • Rewrite to FVCOM input format and save to casename_wnd.dat.
        • model graph: View
        • coverage: from 1979.1.1 to 2014.10.2
        • average duration time of model run: depends on the inputted time scope
  • Cloud Resolving Model (CRM)
    • model introduction: CRM, which explicitly resolve the cloud-scale dynamical circulations and directly couple these physical processes, provide a useful tool to simulate the response of convection and clouds to the large-scale forcing under different climate regimes in producing cloud, precipitation and radiative properties. CRM simulations validated by field experiments have shown great values to improving the process-level understanding of cloud-related processes and their representation (such as closure assumption, trigger condition, convective momentum transport, and subgrid cloud distribution) in general circulation models (GCMs).
    • The preprocess models for CRM include:
      • Derived data of CSU array-averaged analysis products
        • explanation: These data products were created by objectively analyzing upper-air sonde data onto a regular 1-degree, 25-hPa grid using multiquadric interpolation. The gridded data fields were then averaged over the Northern Sounding Array (NSA) and Southern Sounding Arrays (SSA). The files include:
          • Initial sounding
          • Velocity profile file
          • Profiles for large scale forcing terms
          • Time series of ocean surface potential temperature and water vapor mixing ratio
          • Temperature and water vapor mixing ratio profiles for selected period
        • model graph: View
        • coverage: 1 October - 31 Decemeber 2011, 6-hr resolution
        • average duration time of model run: less than 3 minutes
  • Community Multi-scale Air Quality Model (CMAQM)
    • model introduction: The Community Multiscale Air Quality Model (CMAQ) is an active open-source community-developed air quality model that consists of a suite of programs for conducting air quality model simulations. It is a widely used regional chemical transport model, providing technically sound estimates of ozone, particulates, toxics, and acid deposition. CMAQ has arguably the largest user community around the world.

4. Design&Run Preprocess Models

Design new models - VDP (Virtual data product) designer

  • Intro: VDP is an abstract model designer for generating virtual data product. Virtual data product means the product which doesn't exist before users request them. Once a request is initialized, the product will be generated dynamically by running a data generation model. VDP is developed by Center for Spaital Information Science and Systems, George Mason University. The website of VDP is http://www3.csiss.gmu.edu/VDP. Currently, the website offers VDP in several versions which can be used in both Microsoft Windows and Linux systems.
  • Usage: The tool provides an interface for experts to design a new model by dragging and dropping the function component from a service tree to a workspace canvas and connecting the services by sequencely doubleclicking two objects on the workspace canvas. VDP will validate the designed models and lead experts to a pop-up wizard which lets them complete the establishment of models and related virtual products. The models and products will be all registered into our catalog. VDP also provides a button to automatically instantiate the abstract model into a runnable workflow instance in a standard workflow language (such as BPEL, Business Process Execution Language).

Run models - BPELPower

  • Intro: BPELPower is a workflow engine software developed by Center for Spatial Information Science and Systems, George Mason University. It support parsing and executing WS-BPEL (1.1), WSDL 1.1, SOAP (1.1, 1.2), WS-Addressing (2003/03/03,2004/04/08, 2005/05/08), WS-Security (WS-Policy 1.2, SAML 1.1). BPELPower is used to execute the generated workflow instances to complete the model runs. The current address of BPELPower is http://www3.csiss.gmu.edu/BPELPower.
  • Usage: The workflow package will be uploaded to BPELPower for deployment. Once the deployment is done, BPELPower will offer a port for users to invoke the deployed workflow. If users input corresponding values to the exposed input parameters of the workflow, the workflow instance will be activated to run. When the workflow run is over, the output will be return to the users who initiated the execution.

5. Register new web services/Unregister web services

Register new web services

  • This function provides an interface to register new services into our catalog. Once a service is registered, you will be able to find it in the service tree of VDP. Currently, the registration function only supports WSDL. The services with other description language and regulations are not allowed for now. Make sure the inputted WSDL URLs are accessible. Otherwise, the registration will fail.

Un-register old web services

  • This function is provided to remove those web services which are no longer available or used. For now, WSDL only.

6. Order Submission and Status Checking

Order status checking

  • CyberConnector implementation uses an order mechanism. Every user request is considered as an order. The orders are first stored in a table and wait to be processed. If a worker thread in CyberConnector is available, a new order will be assigned to the worker to run instantly. Because the number of worker threads is restricted, not every order will be processed instantly once they are submitted. If the available workers are zero, the new-submitted orders will have to wait until a worker is released. So the duration of processing each order varies according to order waiting time and server load.
  • Users can check the status of their orders through this interface. Everytime users submit a request, an order id will be assigned and sent to users' email account. Using the order id users can query the status of the corresponding order.
